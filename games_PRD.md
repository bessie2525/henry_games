# 舒尔特类认知训练小游戏网站 PRD

> **文档目标**：定义一个面向反应能力、记忆能力、视觉能力训练的小游戏网站 PRD，覆盖黑猩猩测试、数字记忆、序列记忆、视觉记忆四个核心游戏。Codex 或开发者可依据本文直接生成可运行的网站前端与基础数据逻辑。

## 1. 项目概述

### 1.1 背景

用户希望构建一个类似 Human Benchmark 与舒尔特练习体验的认知训练小游戏网站，通过短时、可重复、低门槛的交互任务训练反应速度、工作记忆、视觉搜索、空间记忆与序列记忆能力。

网站首期包含四个游戏：黑猩猩测试、数字记忆、序列记忆、视觉记忆。所有游戏均支持用户在开始前人工选择起始关卡难度，而不是必须从第 1 关开始。

### 1.2 产品目标

| 目标类型 | 说明 | 衡量方式 |
|---|---|---|
| 可玩性 | 四个游戏均可独立开始、暂停退出、失败重开，并具备清晰的关卡反馈。 | 用户无需说明即可完成一次完整游戏流程。 |
| 可配置性 | 每个游戏支持选择起始关卡难度，便于跳过过低难度。 | 游戏开始前可选择难度，并按所选难度初始化。 |
| 训练反馈 | 每轮游戏展示当前关卡、得分、正确率、最好成绩等反馈。 | 结束页可查看本轮结果与历史最佳。 |
| 可扩展性 | 游戏规则、关卡参数、得分逻辑独立配置，便于后续新增游戏。 | 新增游戏时无需重写全站框架。 |

## 2. 目标用户与使用场景

### 2.1 目标用户

| 用户类型 | 核心诉求 | 产品机会 |
|---|---|---|
| 普通训练用户 | 想通过小游戏训练注意力、记忆力和反应能力。 | 提供轻量化、即时反馈、可重复挑战的训练体验。 |
| 挑战型用户 | 希望不断突破个人最好成绩。 | 提供历史最佳、连续通关数、最高关卡记录。 |
| 高阶用户 | 不希望从低难度开始，想直接挑战高难度。 | 提供人工选择起始关卡难度能力。 |

### 2.2 典型使用场景

1. 用户在碎片时间打开网站，选择一个小游戏进行 1 到 3 分钟训练。
2. 用户选择从中高难度开始，快速进入挑战状态。
3. 用户失败后查看成绩反馈，并选择重玩当前难度或回到游戏首页。
4. 用户多次训练后查看本地保存的最佳成绩。

## 5. 通用功能需求

### 5.1 游戏通用状态机

| 状态 | 说明 | 允许操作 |
|---|---|---|
| `intro` | 游戏说明与起始难度选择。 | 调整难度、开始游戏、返回首页。 |
| `ready` | 游戏已初始化，等待用户开始或短暂倒计时。 | 开始、返回。 |
| `showing` | 系统展示需要记忆的内容。 | 通常不允许点击，或点击无效。 |
| `input` | 用户进行点击或输入。 | 提交答案、撤销、重置当前输入。 |
| `success` | 当前关卡通过。 | 进入下一关、继续挑战。 |
| `failed` | 当前关卡失败，游戏结束或扣除生命。 | 重玩、调整难度、返回首页。 |
| `result` | 展示本轮最终成绩。 | 再玩一次、回首页、切换游戏。 |

### 5.2 起始难度选择

> **强需求**：每个游戏必须在开始前提供人工选择起始关卡难度的能力。用户选择后，游戏应直接按该难度初始化，不再强制从最低难度开始。

| 游戏 | 难度含义 | 默认值 | 可选范围建议 |
|---|---|---|---|
| 黑猩猩测试 | 需要记忆并顺序点击的数字数量。 | 4 | 4 到 20 |
| 数字记忆 | 需要记忆的数字位数。 | 3 | 3 到 20 |
| 序列记忆 | 需要复现的亮灯序列长度。 | 1 | 1 到 30 |
| 视觉记忆 | 需要记忆的目标格子数量与网格规模的综合关卡。 | 1 | 1 到 20 |

### 5.3 成绩与本地存储

- 每个游戏独立保存最高成绩。
- 使用 `localStorage` 保存，无账号时不依赖后端。
- 保存字段包括游戏 ID、最高关卡、最高分、最佳正确率、最近游玩时间。
- 结果页如果刷新个人最好成绩，应给出明显提示。

### 5.4 响应式与可访问性

- 桌面端游戏区域建议最大宽度 960px，居中展示。
- 移动端格子尺寸根据视口自适应，避免横向滚动。
- 按钮与格子点击区域不小于 44px。
- 颜色反馈需配合文本或图形状态，不能只依赖颜色区分。
- 支持键盘操作的基础可访问性：Tab 聚焦按钮、Enter 确认输入。

## 6. 游戏一：黑猩猩测试

### 6.1 游戏目标

训练用户的视觉定位、短时空间记忆与顺序回忆能力。系统在网格中随机展示若干数字，用户需要记住数字位置，并按从小到大的顺序点击。

### 6.2 核心规则

1. 根据当前关卡生成 N 个数字，数字从 1 到 N。
2. 数字随机分布在游戏区域内，不能重叠，需保证可点击。
3. 游戏开始时显示所有数字。
4. 当用户点击数字 1 后，其余未点击数字应隐藏为统一方块，仅保留位置。
5. 用户必须按 1、2、3 到 N 的顺序点击。
6. 点击正确则继续；点击错误则当前轮失败。
7. 全部点击正确后进入下一关，下一关 N 增加 1。

### 6.3 起始难度配置

| 参数 | 说明 | 建议值 |
|---|---|---|
| `startLevel` | 起始关卡。 | 默认 1，可选 1 到 17。 |
| `startCount` | 起始数字数量。 | `startCount = startLevel + 3`，因此第 1 关为 4 个数字。 |
| `maxCount` | 最大数字数量。 | 20，可配置。 |
| `gridArea` | 数字分布区域。 | 桌面端约 720px × 480px，移动端自适应。 |

### 6.4 交互流程

1. 用户进入黑猩猩测试页。
2. 页面展示游戏说明、当前最好成绩、起始关卡选择器。
3. 用户选择起始关卡并点击开始。
4. 系统生成对应数量的数字并展示。
5. 用户点击数字 1 后，其余数字变为无数字方块。
6. 用户按顺序点击所有方块。
7. 若成功，展示短暂成功提示并进入下一关。
8. 若失败，展示结果弹窗。

### 6.5 失败与结果

- 失败条件：点击的方块不是当前期望数字。
- 结果展示：达到的最高关卡、最高数字数量、是否刷新最佳成绩。
- 操作按钮：再玩一次、从当前起始难度重玩、调整难度、返回首页。

## 7. 游戏二：数字记忆

### 7.1 游戏目标

训练短时记忆、数字编码能力与注意力保持能力。系统展示一串数字，用户在数字消失后输入刚才看到的数字。

### 7.2 核心规则

1. 当前关卡生成一串长度为 N 的随机数字。
2. 数字展示一段时间后自动隐藏。
3. 用户在输入框中输入记忆的数字。
4. 输入完全一致则通过当前关卡，并进入 N + 1 位数字。
5. 输入不一致则失败并进入结果页。

### 7.3 起始难度配置

| 参数 | 说明 | 建议值 |
|---|---|---|
| `startDigits` | 起始数字位数。 | 默认 3，可选 3 到 20。 |
| `displayDuration` | 数字展示时长。 | `max(1200ms, N × 450ms)`，可配置。 |
| `inputMode` | 输入方式。 | 文本输入框，移动端使用数字键盘。 |
| `allowedChars` | 允许输入字符。 | 0 到 9。 |

### 7.4 交互流程

1. 用户进入数字记忆页。
2. 页面展示说明、最佳位数、起始位数选择器。
3. 用户点击开始。
4. 页面居中展示随机数字，同时可展示倒计时进度条。
5. 展示结束后数字隐藏，输入框自动聚焦。
6. 用户输入答案并提交。
7. 系统判断是否完全一致。
8. 正确则进入下一位数；错误则展示正确答案与用户答案。

### 7.5 结果反馈

- 成功反馈：绿色提示、当前位数 +1。
- 失败反馈：展示正确答案、用户输入、最长记忆位数。
- 得分建议：`score = 当前成功记忆的最大位数`。

## 8. 游戏三：序列记忆

### 8.1 游戏目标

训练用户的顺序记忆、工作记忆与注意力追踪能力。系统在九宫格或更大网格中依次高亮若干格子，用户需要按相同顺序点击复现。

### 8.2 核心规则

1. 页面展示固定网格，首期建议使用 3 × 3 九宫格。
2. 系统生成长度为 N 的随机格子序列，同一个格子允许重复出现。
3. 系统按顺序高亮每个格子。
4. 展示完成后，用户按同样顺序点击格子。
5. 每一步点击正确则继续，错误则失败。
6. 成功复现完整序列后进入下一关，序列长度 N + 1。

### 8.3 起始难度配置

| 参数 | 说明 | 建议值 |
|---|---|---|
| `startSequenceLength` | 起始序列长度。 | 默认 1，可选 1 到 30。 |
| `gridSize` | 网格规模。 | 首期固定 3 × 3；后续可扩展为 4 × 4。 |
| `highlightDuration` | 单个格子高亮时长。 | 500ms。 |
| `highlightGap` | 两个高亮之间的间隔。 | 180ms。 |

### 8.4 交互流程

1. 用户进入序列记忆页。
2. 用户选择起始序列长度并点击开始。
3. 系统生成序列并逐个高亮格子。
4. 展示期间禁用用户点击，避免误操作。
5. 展示结束后进入用户输入阶段。
6. 用户按顺序点击格子，页面显示当前进度。
7. 全部正确则进入下一关；任一步错误则失败。

### 8.5 结果反馈

- 显示最高成功序列长度。
- 显示失败发生在第几步。
- 可提供“观看正确序列回放”按钮，首期可选。

## 9. 游戏四：视觉记忆

### 9.1 游戏目标

训练空间视觉记忆与图形定位能力。系统短暂展示一组目标格子，隐藏后用户需要点击刚才出现过的所有目标格子。

### 9.2 核心规则

1. 当前关卡生成一个网格，并随机选择若干目标格子。
2. 目标格子高亮展示一段时间。
3. 展示结束后所有格子恢复为统一样式。
4. 用户点击自己记住的目标格子。
5. 点击正确目标格子则标记为已选；点击非目标格子则失败。
6. 当所有目标格子都被正确选中后进入下一关。

### 9.3 难度曲线建议

| 关卡范围 | 网格规模 | 目标格子数 | 展示时长 |
|---|---|---|---|
| 1 到 3 | 3 × 3 | 3 到 4 | 1600ms |
| 4 到 7 | 4 × 4 | 4 到 6 | 1500ms |
| 8 到 12 | 5 × 5 | 6 到 8 | 1400ms |
| 13 到 20 | 6 × 6 | 8 到 12 | 1300ms |

### 9.4 起始难度配置

| 参数 | 说明 | 建议值 |
|---|---|---|
| `startLevel` | 起始关卡。 | 默认 1，可选 1 到 20。 |
| `gridRows` | 根据关卡计算出的行数。 | 3 到 6。 |
| `gridCols` | 根据关卡计算出的列数。 | 3 到 6。 |
| `targetCount` | 目标格子数量。 | 随关卡递增。 |
| `previewDuration` | 目标展示时间。 | 1300ms 到 1600ms。 |

### 9.5 交互流程

1. 用户进入视觉记忆页。
2. 选择起始关卡并点击开始。
3. 系统根据关卡生成网格与目标格子。
4. 目标格子短暂高亮展示。
5. 展示结束后目标隐藏，用户开始点击。
6. 用户点击所有记住的目标格子。
7. 若全部正确，进入下一关；若点错，进入失败结果页。

## 10. 统一组件与交互规范

### 10.1 组件清单

| 组件 | 用途 | 关键要求 |
|---|---|---|
| `GameCard` | 首页游戏入口卡片。 | 展示名称、简介、训练能力标签、最佳成绩。 |
| `DifficultySelector` | 起始难度选择。 | 支持滑杆、加减按钮、直接输入；需限制最小最大值。 |
| `GameHeader` | 游戏页顶部状态。 | 展示当前关卡、最好成绩、返回入口。 |
| `ResultModal` | 游戏结束反馈。 | 展示本轮成绩、历史最佳、操作按钮。 |
| `GridBoard` | 格子类游戏区域。 | 支持不同网格规模、点击状态、禁用状态。 |
| `ProgressBar` | 展示倒计时或输入进度。 | 数字记忆、展示阶段可使用。 |

### 10.2 视觉风格

- 整体风格：简洁、科技感、轻量训练工具感。
- 主色建议：深蓝或紫色系，用于按钮、选中态、高亮态。
- 背景建议：浅色模式优先，支持后续扩展深色模式。
- 游戏区域需突出，减少无关装饰。
- 成功、失败、警告状态要有明确颜色与文案反馈。

### 10.3 动效规范

- 按钮点击使用 80 到 150ms 的轻量缩放或明暗反馈。
- 格子高亮使用渐入渐出，避免过度动画影响记忆。
- 失败反馈可使用短暂抖动，但不应超过 300ms。
- 关卡切换使用简短过渡，确保游戏节奏流畅。

- 推荐使用 React + TypeScript + Vite。
- 样式可使用 Tailwind CSS 或 CSS Modules。
- 状态管理首期可使用 React hooks；无需引入复杂全局状态库。
- 路由可使用 React Router，或单页内部状态切换。

- 所有随机生成逻辑应封装在独立 `logic.ts` 中，便于测试。
- 黑猩猩测试的数字位置需要避免重叠，边界内可点击。
- 视觉记忆的目标格子不能重复。
- 序列记忆允许重复格子。
- 数字记忆首位可以为 0，但展示和答案必须保持字符串，不要转为 number 造成前导 0 丢失。

## 13. 验收标准

### 13.1 全局验收

- [ ] 首页展示四个游戏入口，点击后进入对应游戏页。
- [ ] 每个游戏页都有规则说明与起始难度选择器。
- [ ] 用户选择起始难度后，游戏按该难度直接开始。
- [ ] 每个游戏失败后均展示结果，并提供再玩一次、调整难度、返回首页。
- [ ] 刷新页面后，本地最佳成绩仍然保留。
- [ ] 桌面端与移动端均能正常游玩，不出现关键元素溢出。

### 13.2 分游戏验收

| 游戏 | 验收点 | 通过标准 |
|---|---|---|
| 黑猩猩测试 | 按顺序点击数字。 | 点击 1 后隐藏其他数字；顺序正确通关，顺序错误失败。 |
| 数字记忆 | 记忆并输入数字。 | 输入完全一致通关；前导 0 不丢失；错误时展示正确答案。 |
| 序列记忆 | 复现高亮序列。 | 系统播放期间不能输入；用户按完整顺序点击才通关。 |
| 视觉记忆 | 记住目标格子。 | 点中全部目标通关；点击非目标立即失败。 |

### 13.3 边界条件验收

- 起始难度输入低于最小值时，应自动修正为最小值或展示错误提示。
- 起始难度输入高于最大值时，应自动修正为最大值或展示错误提示。
- 快速连续点击不应导致状态错乱或重复进入下一关。
- 游戏展示阶段点击不应产生有效输入。
- 页面刷新或返回首页后，再次进入游戏应回到 `intro` 状态。

## 14. Codex 实现提示

> Codex 生成网站时，应优先保证游戏规则、状态机与起始难度选择正确，再做视觉美化。四个游戏建议共用通用布局与结果弹窗，但各自保留独立逻辑文件。

### 14.1 开发顺序建议

1. 搭建项目框架、首页、路由与基础样式。
2. 实现 `useBestScores` 与 `localStorage` 存储。
3. 实现 `DifficultySelector`、`GameHeader`、`ResultModal`、`GridBoard` 通用组件。
4. 依次实现黑猩猩测试、数字记忆、序列记忆、视觉记忆。
5. 补充移动端适配与边界条件处理。
6. 根据验收标准逐项测试。

### 14.2 必须避免的问题

- 不要把数字记忆的答案转成 number，应始终使用 string。
- 不要让用户在 `showing` 阶段提前输入。
- 不要让难度选择只改变 UI，而没有真正改变游戏初始化参数。
- 不要把四个游戏的逻辑全部写在 `App.tsx` 中，应拆分模块。
- 不要依赖后端服务作为首期必需能力。
